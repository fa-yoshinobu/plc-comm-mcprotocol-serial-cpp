# QJ71C24N RS-232C Format5 Q/L Report

Date: `2026-04-11`

Audience: maintainers validating the `QJ71C24N` path under the same serial settings used for the
later `Format5 / Binary` rechecks.

## Target Setup

- PLC CPU: Mitsubishi Q-series `Q06UDVCPU`
- Serial module: `QJ71C24N`
- Link: `RS-232C`
- Host device: Linux PC with `/dev/ttyUSB0`
- Communication settings: `28800 bps / 8E2 / station 0`
- MC protocol settings: `Format 5 / Binary / sum-check on`
- CLI family selection: `--series ql`

- `read-bits M100 1`: pass

Interpretation:

- the same `Format5 / Binary / 28800 / 8E2` transport works on this Q-series target
- the practical CLI family selection is `--series ql`

## Supported-device Soak

Command:

```bash
MCPROTOCOL_BAUD=28800 \
MCPROTOCOL_DATA_BITS=8 \
MCPROTOCOL_STOP_BITS=2 \
MCPROTOCOL_PARITY=E \
MCPROTOCOL_FRAME=c4-binary \
MCPROTOCOL_SUM_CHECK=on \
MCPROTOCOL_STATION=0 \
MCPROTOCOL_SERIES=ql \
MCPROTOCOL_RESPONSE_TIMEOUT_MS=8000 \
MCPROTOCOL_INTER_BYTE_TIMEOUT_MS=1000 \
./examples/linux_cli/supported_device_rw_soak.sh
```

Observed result:

- run: `pass cycles=14 checks=340 elapsed_sec=180`
- total command count: `1701`
- protocol failures observed during the run: none

Interpretation:

- the no-wait, strictly serial contiguous soak is stable on `QJ71C24N` under `--series ql`
- the observed `verify-mismatch` lines are compatible with RUN-mode overwrite and did not coincide
  with PLC protocol errors

## Native Unresolved Command Recheck

Observed result:

- `0403 random-read`: PLC error `0x7F23`
- `1402 random-write-words`: PLC error `0x7F23`
- `1402 random-write-bits`: PLC error `0x7F23`
- post-read after both `1402` probes still showed no write effect at `D300..D305` and `M300..M305`
- `0801 probe-monitor`: `probe-monitor: skip register 0x7F23`
- `0406 probe-multi-block`: `multi-block-read=skip 0x7F23`
- `1406 probe-multi-block`: `multi-block-write=skip 0x7F22`
- `probe-multi-block` contiguous restore: pass

Interpretation:

- the unresolved native families stay unresolved on this Q-series module as well
- `QJ71C24N + ql` does not turn native random, monitor, or multi-block paths into working command
  families
- the observed PLC error mix matches the `LJ71C24 + ql` setup closely

## Buffer and Qualified Access

Observed result:

- `probe-host-buffer`: `ok`, start `0`, value `0x0000`
- `probe-write-host-buffer`: `skip verify-mismatch expected=0x0001 read=0x0000`
- `probe-module-buffer`: `ok`, start `0`, bytes `A0 B5`
- `probe-write-module-buffer`: `ok A0 B5 -> F5 1F -> A0 B5`
- helper `read-qualified-words 'U3E0\G10' 1`: `0x1000`
- helper `read-qualified-words 'U3E0\HG20' 1`: initial value `0x01E0`
- helper `write-qualified-words 'U3E0\HG20' 0x1234`: write/readback/restore all passed
- native `read-native-qualified-words 'U3E0\G10' 1`: PLC error `0x4030`
- native `write-native-qualified-words 'U3E0\G10' 0x1234`: PLC error `0x4030`
- native `U3E0\HG20` read/write under `--series ql`: rejected before TX because `HG` requires iQ-R

Interpretation:

- helper qualified access remains the practical `U...` path here as well
- helper `U3E0\G10` is neither the `RJ71C24-R2` helper value (`0x83BD`) nor the `LJ71C24` helper
  value (`0x0000`), so qualified helper observations are module-dependent and should stay documented
  per hardware target
- module-buffer access is stronger than host-buffer write access on this setup
- native qualified `U3E0\G10` is still unresolved and again fails with `0x4030`
