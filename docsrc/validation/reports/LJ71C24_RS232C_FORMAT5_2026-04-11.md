# LJ71C24 RS-232C Format5 Q/L Report

Date: `2026-04-11`

Audience: maintainers validating the `LJ71C24` path after switching the hardware away from the
earlier `RJ71C24-R2` iQ-R setup.

## Target Setup

- PLC CPU: Mitsubishi L-series `L26CPU-BT`
- Serial module: `LJ71C24`
- Link: `RS-232C`
- Host device: Linux PC with `/dev/ttyUSB0`
- Communication settings: `28800 bps / 8E2 / station 0`
- MC protocol settings: `Format 5 / Binary / sum-check on`
- CLI family selection: `--series ql`

## Series Selection Check

Commands:

```bash
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series ql cpu-model
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series ql read-words D100 1
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series ql read-bits M100 1
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr read-words D100 1
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr read-bits M100 1
```

Observed result:

- `cpu-model`: `L26CPU-BT`, `0x0542`
- `--series ql read-words D100 1`: pass
- `--series ql read-bits M100 1`: pass
- `--series iqr read-words D100 1`: PLC error `0x7F22`
- `--series iqr read-bits M100 1`: PLC error `0x7F22`

Interpretation:

- the `LJ71C24` setup itself is not the reason contiguous access fails under `--series iqr`
- the CPU family selection must match the actual PLC family
- for `L26CPU-BT`, the practical CLI setting is `--series ql`

## Supported-device Screening

Observed result:

- non-low-address screening using `STS10`, `STN10`, `D100`, `R100`, and the rest of the supported soak set:
  `ok=25 fail=0`
- bit families often showed `verify-mismatch`
- word families typically read back `0x0000 -> 0x0001 -> 0x0000`

Interpretation:

- the supported contiguous `0401/1401` path is usable on this setup
- bit-family readback can be overwritten again while RUN is active, so the soak treats that as
  observation data rather than a protocol failure

## Supported-device Soak

```bash

MCPROTOCOL_DATA_BITS=8 \
MCPROTOCOL_STOP_BITS=2 \
MCPROTOCOL_PARITY=E \
MCPROTOCOL_FRAME=c4-binary \
MCPROTOCOL_SUM_CHECK=on \
MCPROTOCOL_STATION=0 \
MCPROTOCOL_SERIES=ql \
MCPROTOCOL_RESPONSE_TIMEOUT_MS=8000 \
MCPROTOCOL_INTER_BYTE_TIMEOUT_MS=1000 \
./examples/linux_cli/supported_device_rw_soak.sh
```

Observed result:

- first run: `pass cycles=13 checks=325 elapsed_sec=180`
- first-run total command count: `1626`
- second run: `pass cycles=14 checks=326 elapsed_sec=180`
- second-run total command count: `1631`
- protocol failures observed during either run: none

Interpretation:

- the no-wait, strictly serial contiguous soak is stable on `LJ71C24` when `--series ql` is used
- the observed `verify-mismatch` lines are compatible with RUN-mode overwrite and did not coincide
  with PLC protocol errors

## Native Command Recheck

Observed result on the corrected encoder:

- `probe-random-read`: word/bit `single`, `dense`, and `sparse` all `ok native`
- `probe-random-write-words`: `single`, `dense`, and `sparse` all `ok native`, `restore=ok`
- `probe-random-write-bits`: `single`, `dense`, and `sparse` all `ok native`, `restore=ok`
- `probe-multi-block[mixed]`: `read=native write=native restore=ok`
- `probe-monitor=ok mode=native D100=0 D105=0 M100=0 M105=0`
- `probe-monitor[read-only]=ok response=data bytes=8`

Interpretation:

- on `LJ71C24 + L26CPU-BT`, the corrected binary encoder and `--series ql` make the practical native
  random / write / multi-block / monitor families pass
- the older `0x7F23` / `0x7F22` observations should now be treated as pre-fix results, not as the
  current validation state

## Buffer and Qualified Access

Observed result:

- `probe-host-buffer`: `ok`, start `0`, value `0x0000`
- `probe-write-host-buffer`: `skip verify-mismatch expected=0x0001 read=0x0000`
- `probe-module-buffer`: `ok`, start `0`, bytes `ED 19`
- `probe-write-module-buffer`: `ok ED 19 -> B8 B3 -> ED 19`
- helper `read-qualified-words 'U3E0\G10' 1`: `0x0000`
- helper `read-qualified-words 'U3E0\HG20' 1`: `0x0000`
- helper `write-qualified-words 'U3E0\HG20' 0x1234`: write/readback/restore all passed
- native `read-native-qualified-words 'U3E0\G10' 1`: PLC error `0x4030`
- native `write-native-qualified-words 'U3E0\G10' 0x1234`: PLC error `0x4030`
- native `U3E0\HG20` read/write under `--series ql`: rejected before TX because `HG` requires iQ-R

Interpretation:

- helper qualified access remains the practical `U...` path here as well
- module-buffer access is stronger than host-buffer write access on this setup
- native qualified `U3E0\G10` is still unresolved and now fails with `0x4030`
- native `HG` access is not applicable to the L-series `ql` path
