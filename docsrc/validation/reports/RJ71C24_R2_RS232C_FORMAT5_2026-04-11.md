# RJ71C24-R2 RS-232C Format5 Bootstrap Report

Date: `2026-04-11`

Audience: maintainers who need the first real-hardware check after switching the C24 settings away from
the previous Format4 ASCII setup.

## Target Setup

- PLC CPU: Mitsubishi iQ-R `R08CPU`
- Serial module: `RJ71C24-R2`
- Link: `RS-232C`
- Host device: Linux PC with `/dev/ttyUSB0`
- Communication settings: `28800 bps / 8E2 / station 0`
- MC protocol settings: `Format 5 / Binary / sum-check on`

## Scope Note

This report is not directly comparable to the existing `Format 4 / ASCII / 19200 / 8E1 / sum-check off`
reports. The C24 communication mode was changed before these probes.

## Summary

| Category | Status |
|---|---|
| host support | added Linux custom-baud handling so `28800` opens correctly |
| binary transport | fixed partial-frame handling for chunked Format5 responses |
| link bootstrap | `cpu-model` passed |
| contiguous read | `read-words D100 2` passed |
| qualified helper read | `U3E0\\G10=0x83BD`, `U3E0\\HG20=0x0000` |
| native qualified read | `U3E0\\G10` returned success with `0x0000`, `U3E0\\HG20` returned `0x4031` |
| native random read | `0403 0002` returned `0x7F23` for both `D-only` and mixed `D+M` probes |
| native random write | `1402 0002/0003` returned `0x7F23` for both word and bit probes; no write effect confirmed |
| native multi-block | `0406 0002` returned `0x7F23`, `1406 0002` returned `0x7F22`, restore passed |
| native monitor register | `0801 0002` returned `0x7F23` |
| C24-side status words | `U3E0\\G599`, `U3E0\\G600`, and `U3E0\\G31998..32003` stayed `0x0000` even after `0x7F23` and `0x4031` probes |

## Host-side Fixes Required

- `src/posix_serial.cpp`: Linux custom baud support was required because the local libc did not expose `B28800`
- `src/client.cpp`: stream decode now keeps an incomplete leading binary frame instead of discarding it when a later
  `0x10` byte appears inside the partial payload

## Qualified Read Recheck

Commands:

```bash
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on read-qualified-words 'U3E0\G10' 1
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on read-qualified-words 'U3E0\HG20' 1
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --response-timeout-ms 8000 --inter-byte-timeout-ms 1000 --dump-frames on read-native-qualified-words 'U3E0\G10' 1
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --response-timeout-ms 8000 --inter-byte-timeout-ms 1000 --dump-frames on read-native-qualified-words 'U3E0\HG20' 1
```

Observed result:

- helper `read-qualified-words 'U3E0\G10' 1`: `0x83BD`
- helper `read-qualified-words 'U3E0\HG20' 1`: `0x0000`
- native `read-native-qualified-words 'U3E0\G10' 1`: success with `0x0000`
- repeated native `read-native-qualified-words 'U3E0\G10' 1`: still `0x0000`
- native `read-native-qualified-words 'U3E0\HG20' 1`: PLC error `0x4031`

Interpretation:

- the helper `0601/1601` path still behaves like the practical qualified-device path on the new setup
- native qualified `U3E0\G10` no longer times out on this setup, but it also does not match the helper value
- native qualified `U3E0\HG20` is still unresolved and now fails with a different PLC error family than the old Format4/ASCII setup
- treat the native qualified `U3E0\G10=0x0000` result as an unresolved semantic mismatch, not as validated proof that native qualified access is correct

## Random Read Recheck

Commands:

```bash
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --response-timeout-ms 8000 --inter-byte-timeout-ms 1000 --dump-frames on random-read D100 D105
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --response-timeout-ms 8000 --inter-byte-timeout-ms 1000 --dump-frames on random-read D100 D105 M100 M105
```

Observed result:

- `random-read D100 D105`: PLC error `0x7F23`
- `random-read D100 D105 M100 M105`: PLC error `0x7F23`

Interpretation:

- after switching to the actual Format5/Binary setup, the native `0403` failure mode changed from the old Format4/ASCII
  `0x7F22` observations to `0x7F23`
- the rejection is no longer explained by the old ASCII transport or old serial settings

## Random Write Recheck

Commands:

```bash
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on read-words D300 6
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on random-write-words D300=123 D305=456
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on read-words D300 6
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr write-words D300=0 D305=0
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on read-bits M300 6
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on random-write-bits M300=1 M305=0
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr --dump-frames on read-bits M300 6
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr write-bits M300=0 M305=0
```

Observed result:

- baseline `read-words D300 6` returned all zeroes
- `random-write-words D300=123 D305=456` returned PLC error `0x7F23`
- post-read still showed `D300..D305 = 0`
- helper restore `write-words D300=0 D305=0` succeeded
- baseline `read-bits M300 6` returned all zeroes
- `random-write-bits M300=1 M305=0` returned PLC error `0x7F23`
- post-read still showed `M300..M305 = 0`
- helper restore `write-bits M300=0 M305=0` succeeded

Interpretation:

- under the current Format5/Binary setup, both native `1402` word and bit paths now fail with `0x7F23`
- the previous Format4/ASCII split between `1402 words=0x7F22` and `1402 bits=0x7F23` does not hold on this setup
- repeated readback still does not show any native write effect

## C24-side Status Recheck

Commands:

```bash
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr read-qualified-words 'U3E0\G599' 2
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr random-read D100 D105 M100 M105
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr read-qualified-words 'U3E0\G599' 2
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr read-qualified-words 'U3E0\G31998' 6
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr read-native-qualified-words 'U3E0\HG20' 1
./build/mcprotocol_cli --device /dev/ttyUSB0 --baud 28800 --data-bits 8 --stop-bits 2 --parity E --frame c4-binary --sum-check on --station 0 --series iqr read-qualified-words 'U3E0\G31998' 6
```

Observed result:

- baseline `read-qualified-words 'U3E0\G599' 2`: `U3E0\G599=0x0000`, `U3E0\G600=0x0000`
- after native `0403` returned `0x7F23`, `read-qualified-words 'U3E0\G599' 2` still returned `0x0000`, `0x0000`
- baseline `read-qualified-words 'U3E0\G31998' 6` returned all zeroes
- after native `read-native-qualified-words 'U3E0\HG20' 1` returned `0x4031`, `read-qualified-words 'U3E0\G31998' 6` still returned all zeroes
- after a separate native `0403=0x7F23` recheck, `read-qualified-words 'U3E0\G31998' 6` still returned all zeroes

Interpretation:

- the current `0x7F23` and `0x4031` results are not mirrored into the helper-visible `U3E0\G599/600` status words
- the first visible words of the application-manual MODBUS error log area (`U3E0\G31998..32003`) also stayed zero
- on the current setup, these native failures look more like PLC end-code rejections returned in otherwise normal replies than C24-side transport faults recorded in the exposed status words
