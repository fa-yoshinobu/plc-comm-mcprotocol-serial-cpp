# RJ71C24-R2 RS-232C Format4 Revalidation Report

Date: `2026-04-11`

Audience: maintainers who need the exact evidence from the follow-up native-probe revalidation.

## Target Setup

- PLC CPU: Mitsubishi iQ-R `R08CPU`
- Serial module: `RJ71C24-R2`
- Link: `RS-232C`
- Host device: Linux PC with `/dev/ttyUSB0`
- Communication settings: `19200 bps / 8E1 / station 0`
- MC protocol settings: `Format 4 / ASCII / CRLF / sum-check off`

## Summary

| Category | Status |
|---|---|
| link recheck | `cpu-model` passed again when the UART was drained and used serially |
| helper qualified path | `U3E0\G10=0x83BD`, `U3E0\HG20=0x0000` |
| native ng | `0403=0x7F22`, `1402 words=0x7F22`, `1402 bits=0x7F23`, `0406=0x7F22`, `1406=0x7F22`, `0801=0x7F22` |
| native qualified hold | `U3E0\G10` alternated between `0x7F22` and timeout, `U3E0\HG20` native read returned `0x7F22`, native `U3E0\HG20` write timed out |
| transport recovery | ASCII `EOT CRLF` and `CL CRLF` both restored a clean `cpu-model` probe after timeout / mixed-response states |
| write-effect verification | no native random or native qualified write effect was confirmed |

## Operator Note

This revalidation exposed a tooling constraint on the shared UART:

- run `/dev/ttyUSB0` probes strictly serially
- drain pending RX bytes before each probe
- when a timeout or mixed response is observed, send ASCII `EOT CRLF` or `CL CRLF` before the next probe
- do not trust results from parallel runs, because mixed response fragments were observed

The final observations below were taken with a drain step before each command and no overlapping UART use.


## Qualified Helper Recheck

Commands:

```bash
./build/mcprotocol_cli ... --dump-frames on read-qualified-words 'U3E0\G10' 1
./build/mcprotocol_cli ... read-qualified-words 'U3E0\HG20' 1
```

Observed result:

- `U3E0\G10=0x83BD`
- `U3E0\HG20=0x0000`

Interpretation:

- the helper path over `0601/1601` remains the practical qualified-device path
- helper `U3E0\G10` is no longer supported only by an old failing spot check; it succeeded in this revalidation

## Native Probe Recheck

```bash
./build/mcprotocol_cli ... --dump-frames on random-read D100 D105 M100 M105
./build/mcprotocol_cli ... --dump-frames on probe-monitor
```

Observed result:

- `random-read request failed: PLC returned an error (0x7F22)`
- `probe-monitor: skip register 0x7F22`

### Multi-block

Command:

```bash
./build/mcprotocol_cli ... --dump-frames on probe-multi-block
```

Observed result:

```text
multi-block-read=skip 0x7F22
multi-block-write=skip 0x7F22
probe-multi-block: read=skip write=skip restore=ok
```

### Native qualified reads

Commands:

```bash
./build/mcprotocol_cli ... --series iqr read-native-qualified-words 'U3E0\G10' 1
./build/mcprotocol_cli ... --series iqr --dump-frames on read-native-qualified-words 'U3E0\HG20' 1
```

Observed result:

- `U3E0\G10`: mixed across retries, with both `0x7F22` and timeout observed
- `U3E0\HG20`: native read returned `0x7F22`

Interpretation:

- native extended-device access remains unresolved
- the `U3E0\G10` path is unstable enough that a single probe should not be treated as definitive proof of a working native path

## Native Write Verification

### Random write words

Commands:

```bash
./build/mcprotocol_cli ... --dump-frames on random-write-words D300=123 D305=456
./build/mcprotocol_cli ... read-words D300 6
./build/mcprotocol_cli ... write-words D300=0 D305=0
```

Observed result:

- native write request failed with `0x7F22`
- post-read still showed `D300..D305 = 0`
- restore command succeeded

### Random write bits

Commands:

```bash
./build/mcprotocol_cli ... --dump-frames on random-write-bits M300=1 M305=0
./build/mcprotocol_cli ... --dump-frames on read-bits M300 6
./build/mcprotocol_cli ... write-bits M300=0 M305=0
```

Observed result:

- native bit write request failed with `0x7F23`
- post-read still showed `M300..M305 = 0`
- restore command succeeded

Follow-up request-shape recheck:

- the host-side ASCII encoder was corrected to stop sending single-character bit states for `1402`
- `Q/L` shape now sends `00/01` set-reset bytes in ASCII, matching the MC protocol manual example
- `iQ-R` shape now sends word-sized ASCII set-reset values for subcommand `0003`
- on `2026-04-11`, both shapes still returned `0x7F23` on the validated `RJ71C24-R2` setup

### Native qualified write

Commands:

```bash
./build/mcprotocol_cli ... --series iqr --dump-frames on write-native-qualified-words 'U3E0\HG20' 0x1234
./build/mcprotocol_cli ... read-qualified-words 'U3E0\HG20' 1
./build/mcprotocol_cli ... write-qualified-words 'U3E0\HG20' 0x0000
./build/mcprotocol_cli ... read-qualified-words 'U3E0\HG20' 1
```

Observed result:

- native qualified write timed out
- one immediate helper read after that returned `0x7F22`
- helper restore succeeded
- final helper read returned `U3E0\HG20=0x0000`

Interpretation:

- no validated native qualified write effect was proven
- the helper path remained recoverable after the failed native write attempt

## C24 Recovery Recheck

Commands:

```bash
printf '\004\r\n' > /dev/ttyUSB0
./build/mcprotocol_cli ... --dump-frames on cpu-model
./build/mcprotocol_cli ... --dump-frames on random-read D100 D105 M100 M105
./build/mcprotocol_cli ... --dump-frames on probe-monitor
./build/mcprotocol_cli ... --series iqr --dump-frames on read-native-qualified-words 'U3E0\G10' 1
printf '\004\r\n' > /dev/ttyUSB0
./build/mcprotocol_cli ... --dump-frames on cpu-model
./build/mcprotocol_cli ... recover-c24 cl
./build/mcprotocol_cli ... --dump-frames on cpu-model
```

Observed result:

- ASCII `EOT CRLF` itself returned no payload
- ASCII `CL CRLF` itself returned no payload
- `cpu-model` passed immediately after both `EOT CRLF` and `CL CRLF`
- `0403` random read still returned `0x7F22` after `EOT CRLF`
- `0801` monitor probe still returned `0x7F22` after `EOT CRLF`
- after a native-qualified `U3E0\G10` timeout, both `EOT CRLF` and `CL CRLF` restored a clean `cpu-model` probe

Interpretation:

- `EOT CRLF` and `CL CRLF` are both useful as C24 transmission-sequence recovery after abnormal communication
- neither recovery code converted any unresolved native command family into a native pass

## iQ-R Series Recheck

Commands:

```bash
./build/mcprotocol_cli ... --series iqr --dump-frames on random-read D100 D105 M100 M105
./build/mcprotocol_cli ... --series iqr --dump-frames on probe-monitor
./build/mcprotocol_cli ... --series iqr --dump-frames on probe-multi-block
./build/mcprotocol_cli ... --series iqr --dump-frames on random-write-words D300=123 D305=456
./build/mcprotocol_cli ... --dump-frames on read-words D300 6
./build/mcprotocol_cli ... write-words D300=0 D305=0
```

Observed result:

- `random-read` switched to native `0403 0002` for iQ-R framing and still failed with `0x7F22`
- `probe-monitor` switched to native `0801 0002` and still reported `0x7F22`
- `probe-multi-block` switched to native `0406/1406 0002` and both paths still reported `0x7F22`
- `random-write-words` switched to native `1402 0002` and still failed with `0x7F22`
- `probe-multi-block` backup and restore traffic over contiguous `0401/1401` switched to `0002/0003` and still passed
- post-read after the failed `1402 0002` write still showed `D300..D305 = 0`
- helper restore `write-words D300=0 D305=0` still succeeded

Interpretation:

- omitting `--series iqr` was not the cause of the unresolved native rejections on this setup
- iQ-R series selection changes the transmitted subcommand shape as expected, but it does not rescue `0403`, `1402` words, `0406`, `1406`, or `0801`
- contiguous `0401/1401` traffic still works with iQ-R subcommands, so this is not a global iQ-R framing failure
